# テスト依頼テンプレート（test フェーズ用）

> このテンプレートは `/neko-gundan test` の入力として使うテスト依頼のガイドです。
> フォーマットの完全一致は不要。口頭指示でも動けます。

## 必須項目（これがないとテストを始められない）

### 1. テスト対象
プロジェクト名と対象範囲。
- 例: 「my-app 全体」「my-app の認証機能だけ」「直近のコミットで変更されたファイル」

### 2. テスト種別
指定がなければ**単体テスト+結合テスト（両方必須）**がデフォルト。
- **単体テスト**: 関数・メソッド単位（正常系・異常系は必須）
- **結合テスト**: 機能間連携・API・DB連携（単体テスト後に必須）
- **E2Eテスト**: ブラウザ操作（指定時のみ）
- **セキュリティ**: semgrep, trivy, gitleaks 等（指定時のみ）
- **lint/型チェック**: tsc, ruff, eslint 等（指定時のみ）

## あると助かる項目

### 成功基準
- 例: 「テスト全件通過」「カバレッジ80%以上」「セキュリティスキャンでHigh以上ゼロ」

### 既存テストの有無
テストが既にあるか、新規作成が必要か。

### 重点確認ポイント
「ここが壊れやすい」「ここのエッジケースが心配」等。

---

## テストフェーズの実行フロー

```
1. 対象コードを読む
2. 機能一覧を洗い出す
3. テスト計画書を作成（テストマトリクス含む）
4. 総司令に提示 → 承認
5. 単体テスト作成・実行
6. 結合テスト作成・実行
7. 結果をマトリクスに記録
8. テスト結果報告
```

## テスト計画書（test-plan）

テスト開始前に**テスト計画書**を作成し、総司令の承認を得てからテスト作成に入る。

### テストマトリクス（必須）

機能 × テスト観点のマトリクスで網羅性を担保する。

```markdown
## 単体テストマトリクス

| # | 機能/関数 | 正常系 | 異常系 | 境界値 | 状態遷移 | 備考 |
|---|----------|--------|--------|--------|---------|------|
| 1 | login()  | [ ]    | [ ]    | [ ]    | [ ]     |      |
| 2 | logout() | [ ]    | [ ]    | ―      | [ ]     |      |
| 3 | validate()| [ ]   | [ ]    | [ ]    | ―       |      |

## 結合テストマトリクス

| # | テストシナリオ | 関連機能 | 期待結果 | 状態 | 備考 |
|---|-------------|---------|---------|------|------|
| 1 | ログイン→トークン取得→認証付きAPI呼出 | login, auth middleware | 200 OK | [ ] | |
| 2 | 無効トークンでAPI呼出 | auth middleware | 401 | [ ] | |
```

### テスト観点

| 区分 | 観点 | 必須 | 説明 |
|------|------|------|------|
| 正常系 | 期待通りの入力→期待通りの出力 | **必須** | Happy path |
| 異常系 | 不正入力・null・型違い・未認証・権限外 | **必須** | エラーハンドリングの検証 |
| 境界値 | 0/1/MAX/空文字/空配列/上限値±1 | 推奨 | 該当する場合 |
| 状態遷移 | 状態変化のある機能の前後確認 | 該当時 | ログイン状態の前後等 |

### 結合テスト観点

| 観点 | 説明 | 必須 |
|------|------|------|
| 機能間データフロー | A機能の出力がB機能の入力として正しく渡るか | **必須** |
| エラー伝播 | A機能のエラーがB機能で適切にハンドリングされるか | **必須** |
| 状態の一貫性 | 複数機能をまたいだ状態変化が整合するか | 推奨 |
| 外部依存 | DB・API・ファイルシステム等との連携 | 該当時 |

## 記入例（最低限）

```markdown
# テスト依頼

## 対象
my-app の認証機能（src/auth/）

## 種別
単体テスト + 結合テスト
```

## 口頭指示の例（これでも動ける）

```
/neko-gundan test "my-appの認証機能テストして"
```
→ テスト計画書（マトリクス含む）を作成 → 承認 → 単体テスト → 結合テスト の順で実行
